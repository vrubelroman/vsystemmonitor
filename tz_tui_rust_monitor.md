# Техническое задание  
## TUI-приложение для Linux на Rust для мониторинга системных ресурсов

**Версия документа:** 1.0  
**Статус:** Проект ТЗ для этапа 1 с закладкой архитектуры под этап 2  
**Формат реализации:** TUI (terminal user interface), Linux, Rust

---

## 1. Цель проекта

Разработать TUI-приложение для Linux на языке Rust, предназначенное для визуального мониторинга системных ресурсов.

На **первом этапе** приложение должно отображать метрики **локальной машины** в одной колонке интерфейса.

На **втором этапе** приложение должно получить возможность отображать метрики **удалённых хостов**, список которых определяется на основе SSH-конфигурации пользователя, с сохранением единой архитектуры, удобного UI и возможности масштабирования.

---

## 2. Общая концепция

Приложение должно быть:

- визуально приятным;
- цветным;
- быстрым;
- удобным для работы в терминале;
- архитектурно подготовленным к расширению без серьёзной переработки кода.

Интерфейс должен быть ориентирован на панельный просмотр нескольких хостов, где:

- **первая колонка** — всегда локальная машина;
- **следующие колонки** — удалённые хосты;
- одновременно на экране отображаются **до 3 колонок**;
- если хостов больше 3, пользователь переключает страницы/наборы колонок клавишей;
- механизм переключения колонок должен быть заложен уже на этапе 1, даже если пока отображается только одна колонка.

---

## 3. Этапность разработки

## 3.1. Этап 1 — локальный мониторинг

На первом этапе реализуется:

- запуск приложения на Linux;
- получение и отображение метрик **локальной машины**;
- одна колонка с данными локального хоста;
- конфигурационный файл приложения;
- базовая навигация по интерфейсу;
- подготовленная архитектура под дальнейшее подключение удалённых хостов.

### Обязательные метрики этапа 1

1. **Загрузка CPU**
2. **Использование оперативной памяти**
3. **Занятость диска / дисков**

### Что значит “занятость диска / дисков”

На этапе 1 в ТЗ закладывается поддержка одного из двух режимов:

- либо отображение **общей сводки по файловой системе/основному диску**;
- либо отображение **списка нескольких дисков / mount points**.

**Рекомендуемый вариант:** сразу проектировать UI и внутреннюю модель данных под **список дисков / mount points**, даже если по умолчанию показывается только основной системный раздел. Это упростит развитие без переписывания интерфейса.

---

## 3.2. Этап 2 — удалённые хосты через SSH

Этап 2 не реализуется сейчас, но должен быть явно заложен в архитектуру, конфиг и UI.

На втором этапе приложение должно:

- читать SSH-конфигурацию пользователя;
- извлекать из неё список доступных хостов;
- пытаться подключаться к ним или получать по ним данные;
- отображать метрики локального и удалённых хостов в колонках;
- позволять листать группы колонок, если хостов больше 3.

### Источник хостов

Предполагается использование SSH-конфигурации пользователя, например:

- `~/.ssh/config`

Из него должны читаться записи `Host`, которые пользователь использует как алиасы для SSH-подключений.

### Поведение при отсутствии SSH-конфига

Если SSH-конфиг отсутствует или не содержит пригодных хостов:

- приложение должно работать штатно;
- отображается только **локальная машина**;
- это не считается ошибкой выполнения.

### Поведение при недоступности удалённого хоста

Если удалённый хост не отвечает / недоступен / не пингуется:

- хост не должен ломать интерфейс;
- он должен помещаться в **последнюю отображаемую колонку** или в конец текущего набора колонок;
- в его панели должно быть явно указано состояние вида:
  - `Host unreachable`
  - `No ping`
  - `Connection failed`

**Важно:** на этапе 2 необходимо чётко определить, что именно считается недоступностью:
- ICMP ping не проходит;
- TCP-порт SSH не доступен;
- SSH-аутентификация не удалась;
- таймаут при подключении.

В текущем ТЗ это оставляется как архитектурное требование и открытый вопрос для детализации реализации этапа 2.

## 4. Требования к интерфейсу (UI/UX)

## 4.1. Общая идея интерфейса

Интерфейс должен быть выполнен в виде набора вертикальных колонок.

Каждая колонка представляет **один хост**.

### На этапе 1
- отображается только **1 колонка** — локальная машина.

### На этапе 2
- отображается до **3 колонок одновременно**;
- пользователь может переключать страницы колонок горячими клавишами;
- локальная машина всегда остаётся первым логическим хостом в списке.

---

## 4.2. Структура одной колонки

Каждая колонка должна содержать:

1. **Заголовок хоста**
   - имя хоста;
   - тип хоста:
     - local
     - remote
   - состояние:
     - online
     - unreachable
     - loading
     - error

2. **Блок CPU**
   - текущая загрузка CPU в процентах;
   - желательно визуальная шкала / bar;
   - опционально: среднее значение за короткий интервал.

3. **Блок RAM**
   - используемая память;
   - общая память;
   - процент использования;
   - визуальная шкала / bar.

4. **Блок Disk**
   - использование диска или нескольких дисков;
   - свободно / занято / общий объём;
   - процент использования;
   - визуальная шкала / bar;
   - если дисков несколько — компактный список.

5. **Строка статуса**
   - время последнего обновления;
   - возможная ошибка получения данных;
   - индикатор “данные устарели”, если обновление давно не происходило.

---

## 4.3. Визуальный стиль

Приложение должно выглядеть современно и аккуратно, а не как “сырой системный вывод”.

### Предлагаемый визуальный подход

- тёмная тема по умолчанию;
- контрастные, но не кислотные цвета;
- аккуратные рамки у панелей;
- читабельная типографика для терминала;
- цветовая индикация состояний.

### Предлагаемая цветовая логика

- **Нормальное состояние** — зелёный / cyan
- **Повышенная нагрузка** — жёлтый
- **Критическая нагрузка** — красный
- **Недоступный хост** — серый / красный
- **Заголовки и акценты** — синий / cyan / magenta

### Рекомендуемые пороги подсветки

#### CPU / RAM / Disk
- до 60% — нормальный цвет;
- 60–85% — warning;
- выше 85% — critical.

Пороги желательно вынести в конфиг.

---

## 4.4. Навигация

В интерфейсе должны быть предусмотрены горячие клавиши.

Минимально:

- выход из приложения;
- обновление данных вручную;
- переключение между страницами колонок;
- открытие help / списка клавиш.

### Клавиши

Клавиши должны поддерживать конфигурирование хотя бы частично, в первую очередь:

- клавиша переключения страниц колонок;
- клавиша выхода;
- клавиша ручного обновления;
- клавиша открытия help.

## 5.1. Обязательный параметр из текущего запроса

---

## 5.2. Рекомендуемые дополнительные параметры конфига

Ниже — параметры, которые очень желательно заложить сразу, даже если часть из них будет использоваться позже.

### Общие
- `refresh_interval_ms` — интервал обновления метрик;
- `theme` — тема оформления;
- `show_borders` — показывать ли рамки;
- `compact_mode` — компактный режим интерфейса;
- `show_help_hints` — показывать ли подсказки по клавишам;
- `default_page_size` — сколько колонок показывать одновременно (по умолчанию 3, на этапе 1 фактически 1).

### Горячие клавиши
- `key_next_page`
- `key_prev_page`
- `key_refresh`
- `key_quit`
- `key_help`

### Метрики и отображение
- `cpu_warning_threshold`
- `cpu_critical_threshold`
- `ram_warning_threshold`
- `ram_critical_threshold`
- `disk_warning_threshold`
- `disk_critical_threshold`
- `show_all_disks`
- `disk_include_mountpoints`
- `disk_exclude_mountpoints`

### SSH / удалённые хосты (для этапа 2)
- `ssh_config_path`
- `ssh_connect_timeout_ms`
- `host_ping_timeout_ms`
- `unreachable_to_end` — переносить недоступные хосты в конец;
- `prefer_ssh_over_ping_check` — способ первичной проверки доступности;
- `max_parallel_hosts` — ограничение на число одновременных запросов.

### Поведение
- `sort_hosts` — порядок отображения хостов;
- `show_unreachable_hosts`
- `stale_data_timeout_ms`

---

## 5.3. Рекомендуемый путь конфига

Предлагается поддержка одного из следующих путей:

- `~/.config/<app_name>/config.toml`
- или XDG-совместимый путь

**Рекомендуемый формат файла:** `TOML`

Причины:
- хорошо читается человеком;
- удобно парсится в Rust;
- подходит для структурированных настроек.

---

## 5.4. Пример структуры конфига

```toml
refresh_interval_ms = 1000
theme = "dark"
show_borders = true
compact_mode = false
show_help_hints = true
default_page_size = 3

cpu_warning_threshold = 60
cpu_critical_threshold = 85
ram_warning_threshold = 60
ram_critical_threshold = 85
disk_warning_threshold = 70
disk_critical_threshold = 90

show_all_disks = true
disk_include_mountpoints = ["/", "/home"]
disk_exclude_mountpoints = ["/boot"]

[keys]
next_page = "l"
prev_page = "h"
refresh = "r"
quit = "q"
help = "?"

[ssh]
config_path = "~/.ssh/config"
ssh_connect_timeout_ms = 2500
host_ping_timeout_ms = 1000
unreachable_to_end = true
prefer_ssh_over_ping_check = true
max_parallel_hosts = 8
```

## 6. Функциональные требования

## 6.1. Этап 1

Приложение должно:

1. запускаться в терминале Linux;
2. корректно работать без root-прав;
3. собирать локальные системные метрики;
4. обновлять данные с заданным интервалом;
5. отображать данные в TUI-интерфейсе;
6. читать конфиг при старте;
7. использовать дефолтные значения, если часть параметров отсутствует;
8. не падать при ошибке чтения отдельных метрик;
9. корректно завершаться по горячей клавише;
10. отображать только локальную машину.

---

## 6.2. Архитектурные требования с заделом под этап 2

Архитектура должна разделять:

1. **слой получения данных**  
   Источники метрик должны быть абстрагированы:
   - local collector
   - remote collector

2. **модель данных хоста**  
   Единая структура данных для локального и удалённого хоста:
   - host id
   - display name
   - host type
   - status
   - metrics
   - last update time
   - error state

3. **слой конфигурации**
   - чтение;
   - валидация;
   - подстановка дефолтов.

4. **слой UI**
   - независим от конкретного источника данных;
   - рендерит список хостов/колонок на основе общей модели.

5. **слой навигации / paging**
   - логика переключения “страниц” колонок;
   - независима от того, локальные это хосты или удалённые.

---

## 7. Нефункциональные требования

Приложение должно быть:

- быстрым;
- стабильным;
- понятным;
- расширяемым;
- устойчивым к частичным сбоям;
- пригодным для дальнейшей доработки.

### Требования к производительности
- обновление интерфейса не должно вызывать заметных зависаний;
- приложение не должно чрезмерно нагружать CPU;
- обновления должны выполняться асинхронно или без блокировки UI.

### Требования к отказоустойчивости
- ошибка по одной метрике не должна ломать весь экран;
- ошибка по одному удалённому хосту не должна ломать другие колонки;
- отсутствие SSH-конфига — штатный сценарий.

---

## 8. Логика работы с хостами в будущем (этап 2)

## 8.1. Порядок отображения

Рекомендуемый порядок:

1. локальная машина;
2. доступные удалённые хосты;
3. недоступные удалённые хосты в конце.

Если включена опция `unreachable_to_end`, недоступные хосты всегда сортируются в конец списка.

---

## 8.2. Поведение при большом числе хостов

Если хостов больше 3:

- интерфейс показывает только текущую страницу;
- одна страница содержит до 3 колонок;
- пользователь переключает страницы горячими клавишами.

### Пример
Если хостов 7:

- страница 1: host 1, host 2, host 3
- страница 2: host 4, host 5, host 6
- страница 3: host 7

---

## 8.3. Недоступные хосты

Для недоступного хоста показывается отдельная колонка с:

- именем хоста;
- статусом недоступности;
- краткой причиной;
- временем последней попытки обновления.

Пример содержимого панели:

- `status: unreachable`
- `reason: ssh timeout`
- `last checked: 12:41:03`

## 12. Ограничения и допущения

1. На этапе 1 приложение работает только с локальной машиной.
2. На этапе 2 список хостов берётся из SSH-конфига пользователя.
3. На этапе 2 механизм получения удалённых метрик может быть уточнён отдельно:
   - выполнение удалённой команды по SSH;
   - запуск небольшого агентного скрипта;
   - вызов стандартных системных утилит на удалённой машине.
4. Если SSH-конфиг отсутствует, приложение продолжает работу в локальном режиме.
5. Поддержка Windows и macOS в рамках этого ТЗ не требуется.

---

## 13. Что особенно важно не упустить при проектировании

1. **Не привязывать UI к локальному источнику данных.**  
   Иначе на этапе 2 придётся переписывать интерфейс.

2. **Сразу сделать единый объект “хост”.**  
   Локальный и удалённый хосты должны рендериться одинаково.

3. **Заложить paging даже на этапе 1.**  
   Пусть он пока фактически не нужен, но архитектурно должен существовать.

4. **Конфиг сделать расширяемым.**  
   Чтобы потом не ломать формат файла при добавлении SSH-параметров и клавиш.

5. **Ошибки и недоступность хостов показывать как состояние UI, а не как падение приложения.**

## 16. Критерии готовности этапа 1

Этап 1 считается завершённым, если:

- приложение собирается и запускается на Linux;
- отображает локальную машину в одной колонке;
- показывает CPU, RAM и Disk;
- данные обновляются по таймеру;
- параметры читаются из конфига;
- интерфейс цветной и читаемый;
- код организован так, что к нему можно добавить remote hosts без капитальной переделки.

## 17. Итог

Нужно разработать красивое и расширяемое TUI-приложение на Rust для Linux, которое:

- на **первом этапе** показывает ресурсы **локальной машины**;

- имеет конфиг;
- поддерживает горячие клавиши;
- архитектурно готово к масштабированию;
